走近设计模式（二）：观察者模式

发表于 2018-07-06 更新于 2021-08-18

观察者模式定义了对象之间的一对多依赖，这样一来，当一个对象改变状态时，它的所有依赖者都会受到通知并自动更新

1. 气象监测应用的概况

文中本章开始的时候，描述了一个气象监测的应用场景。

此系统中的三个部分分别是：气象站(获取实际气象数据的物理装置)、WeatherData对象(追踪来着气象站的数据，并更新布告板)、布告板(显示目前天气状况给用户看)。

WeatherData对象知道如何跟物理气象站联系，以获得更新的数据。WeatherData对象会随机更新三个布告板的显示：目前状况(温度、湿度、气压)、气象统计、天气预报。

所以，我们的目标很明确：利用WeatherData对象获得数据，并更新三个布告板。

我们先来看看WeatherData类的设计：其中包括了Temperature、Humidity、Pressure三个状态属性的getter方法，和一个measurementChanged()，一旦气象数据更新了，此方法会被调用。我们的工作就是实现measurementChanged()方法，让它更新布告板。

整理一下思路应该是这样的：

  • WeatherData类具有getter方法，可以取得三个测量值：温度、湿度和气压。
  • 当新的测量数据准备好时，measurementChanged()方法就会被调用。
  • 实现三个使用天气数据的布告板：一旦WeatherData有新的测量，这些布告板必须马上更新。
  • 此系统必须可以扩展，让其他开发人员建立定制的布告板，用户可以随心所欲地添加或删除任何布告板。初始只有这三个布告板：目前状况、天气统计、天气预报。

2. 认识观察者模式

首先我们来了解一下报纸和杂志的订阅是怎么回事：

  • 报社的业务就是出版报纸。
  • 向某家报社订阅报纸，只要他们有新报纸出版，就会给你送来。只要你是他们的客户，你就会一直收到新报纸。
  • 当你不想再看报纸的时候，取消订阅，他们就不会再送新报纸来。
  • 只要报社还在运营，就会一直有人(单位)向他们订阅报纸或取消订阅。

其实：出版者 + 订阅者 = 观察者模式

将出版者称呼为**“主题”(Subject)，订阅者称呼为“观察者”(Observer)**。

让我们再次描述一遍：

  • 主题对象中管理某些数据；
  • 观察者订阅(注册)主题，以便主题数据改变时能收到更新；
  • 当主题内的数据改变，就会通知观察者。
3. 定义观察者模式

当你试图描述观察者模式时，可以使用报纸订阅服务，以及出版者和订阅者比拟这一切。

更加官方一些，观察者模式定义了对象之间的一对多依赖，这样一来，当一个对象改变状态时，它的所有依赖者都会受到通知并自动更新。

4. 松耦合的威力

当两个对象之间松耦合，它们依然可以交互，但是不太清楚彼此的细节。

观察者模式提供了一种对象设计，让主题和观察者之间松耦合。

关于观察者的一切，主题只知道观察者实现了某个接口(也就是Observer接口)。主题不需要知道观察者的具体实现类是谁、做了些什么或者其他的任何细节。

任何时候我们都可以增加新的观察者。因为主题唯一依赖的东西是一个实现Observer接口的对象列表，所以我们可以随时增加观察者。事实上，在运行时我们可以用新的观察者取代现有的观察者，主题不会受到任何影响。同样的，也可以在任何时候删除某些观察者。

在新类型的观察者出现时，主题的代码就不需要修改。假如我们有个新的具体类需要当观察者，我们不需要为了兼容新类型而修改主题的代码，所有要做的就是在新的类里面实现此观察者接口，然后注册为观察者即可。主题不在乎别的，它只会发送通知给所有实现了观察者接口的对象。

我们可以独立地复用主题或观察者。如果我们在其他地方需要使用主题或观察者，可以轻易的复用，因为二者并非紧耦合。

改变主题或观察者其中一方，并不会影响另一方。因为两者是松耦合的，所以只要他们之间的接口仍被遵守，我们就可以自由地改变他们。

设计原则：为了交互对象之间的松耦合设计而努力。

松耦合的设计之所以能让我们建立有弹性的OO系统，能够应对变化，是因为对象之间的互相依赖降到了最低。

6. 使用Java内置的观察模式

目前为止，已经从无到有的完成了观察者模式。其实，Java API有内置的观察者模式。java.uitl包内包含了最基本的Observer接口和Observable类，并且已经内置了许多功能。可以使用push和pull的方式传递数据。

Java 内置的观察者模式是如何运作的呢？

  • 和我们在气象站中的实现类似，但有些小的差异。最明显的差异是WeatherData(主题类)现在扩展自Observable类，并继承了一些增加、删除、通知观察者的方法，以及一些其他方法。

  • 实现观察者接口(java.uitl.Observer)，然后调用任何Observable对象的addObserver()方法，就把对象变成观察者；不想再当观察者时，调用deleteObserver()方法即可。

  • 可观察者(主题)想要发出通知时，首先要调用**setChanged()**方法，标记状态已经改变的事实；然后调用两种notifyObservers()方法中的一个(notifyObservers()和notifyObservers(Object arg))。

    如果想push数据给观察者，可以把数据当做数据对象传递给notifyObservers(Object arg)方法；否则，观察者就必须从主题对象中pull数据。

有必要说一下**setChanged()**方法：

setChanged()方法用来标记状态已经改变的事实，好让notifyObservers()知道当它被调用时，应该更新观察者。如果调用notifyObservers()之前没有先调用setChanged()方法，观察者就不会被通知。

这样做有其必要性。setChanged()方法可以在更新观察者时，有更多的弹性，可以更适当地通知观察者。比如说，如果没有设置setChanged()方法，我们的气象站测量是如此的敏锐，以至于温度计读数每十分之一就会更新，这会造成WeatherData对象持续不断地通知观察者，我们并不希望看到这样的事情发生。如果希望变化半度以上才去更新，就可以在温度差距达到半度的时候，调用setChanged()，进行有效的更新。

8. Observable的黑暗面

应该注意到的是，可观察者是一个“类”，而不是一个“接口”，更糟的是，它甚至没有实现一个接口。

这样一来，会限制它的使用和复用，毕竟如果你的类本来就有父类的话，Java是不支持多继承的。同时Observable将关键的方法保护起来了，setChanged()被定义为protected，这意味着：除非你继承自Observable，否则你无法创建Observable实例并组合到你自己的对象中。

如果你能够扩展Observable类，那么Observable可能会满足你的需求，否则，需要我们自己实现一套观察者模式。其实也没有很难，就像刚刚开始做的那样。

9. 总结

OO原则：为交互对象之间的松耦合设计而努力。

OO模式：观察者模式定义了对象之间的一对多依赖，这样一来，当一个对象改变状态时，它的所有依赖者都会受到通知并自动更新。
